# Validasi Temporal: Menghindari Look-Ahead Leakage

Menerapkan *cross-validation* acak standar (seperti *K-Fold*) pada data deret waktu merupakan kesalahan fundamental. Pemisahan acak mengabaikan arah jalannya waktu karena metode ini mengocok baris data sebelum membaginya ke dalam set pelatihan dan validasi. Akibatnya, data dari masa depan dapat masuk ke dalam set pelatihan sementara model diuji pada data dari masa lalu. Dalam skenario peramalan harga saham harian, pendekatan ini akan mengizinkan model berlatih menggunakan kejatuhan harga di hari Jumat untuk memprediksi harga di hari Kamis. Kesalahan ini menghasilkan *look-ahead leakage*, di mana informasi masa depan merembes ke masa lalu dan secara artifisial melambungkan akurasi. Model tampak sempurna saat dievaluasi, namun gagal total di dunia nyata karena telah mengintip jawaban.

Solusi mendasar untuk masalah ini adalah melakukan *split temporal*. Validasi deret waktu mengharuskan set pelatihan secara ketat mendahului set validasi. Jika kita memiliki data historis satu tahun, kita melatih model menggunakan data dari bulan Januari hingga Oktober, lalu mengujinya secara eksklusif pada data bulan November hingga Desember. Pendekatan ini selaras dengan kondisi produksi yang sebenarnya.

Meski *split temporal* sudah diterapkan, ada risiko kebocoran spesifik-modalitas yang tersembunyi akibat proses *windowing*. *Windowing* mengubah deret waktu yang panjang menjadi sampel-sampel observasi pendek yang saling tumpang-tindih. Tumpang-tindih ini berguna untuk menjaga kontinuitas informasi, namun menjadi celah kebocoran di garis batas pemisahan. Sampel terakhir di dalam set pelatihan dan sampel pertama di dalam set validasi sering kali berbagi baris data mentah yang sama di dalam jendela *lookback* mereka.

Sebagai contoh, jika panjang *lookback* adalah tujuh hari, sampel pengujian pertama di hari ke-100 menggunakan data mentah dari hari ke-93 hingga hari ke-99 sebagai fitur input. Masalahnya, baris data dari rentang waktu yang sama juga digunakan untuk membentuk fitur dan target pada sampel-sampel terakhir di set pelatihan. Jika kita membelah data tepat di antara hari ke-99 dan hari ke-100 tanpa jarak pemisah, model tidak secara langsung melihat nilai target masa depan, tetapi kemiripan representasi input membuat evaluasi menjadi terlalu optimis.

Untuk menegakkan **praktik pipeline yang benar**, kita mencegah kebocoran antar-jendela ini dengan menyisipkan *purge gap* (celah pembersihan) di antara set pelatihan dan set validasi. *Purge gap* adalah area eksklusi tempat kita sengaja membuang sampel di sekitar batas *split temporal*.

Lebar celah ini dihitung secara eksak: jumlah sampel yang dibuang harus sama dengan panjang *lookback* ditambah horizon prediksi. Horizon prediksi wajib ditambahkan karena target dari sampel pelatihan terakhir merentang ke masa depan. Kita melompati rentang waktu tersebut sepenuhnya agar set validasi terbebas dari sisa informasi historis. Dengan menerapkan *purge gap*, kita menjamin sampel pengujian pertama tidak memiliki irisan data mentah dengan sampel pelatihan terakhir. Evaluasi model menjadi ketat dan dapat diandalkan saat menghadapi data yang benar-benar baru.
